# Unified client/provider adaptation surfaces

Research date: 2026-07-16 (America/New_York)

Scope: current public first-party documentation, specifications, and tagged source for Codex, Claude Code, VS Code/GitHub Copilot, and the publicly inspectable ChatGPT/Codex app seam. “Supported” means explicitly documented. Technical feasibility is kept separate from product support and credential authorization. This note does not propose implementation or settle the unified project’s product model.

## Executive findings

1. There is no single “OpenAI-compatible” contract that covers these agent clients. Current Codex speaks Responses only; Claude Code speaks Anthropic Messages or its documented cloud-provider dialects; VS Code has a normalized provider interface and can adapt Chat Completions, Responses, or Messages per model.
2. Tool calls are not interchangeable JSON. The clients differ in tool-name constraints, namespace representation, call/result placement, streaming events, and continuation. Codex currently emits a Responses namespace extension unless the provider capability disables it; public Responses and Messages schemas are flat.
3. Continuation divides into two families: opaque server-side continuation (`previous_response_id` in Responses) and explicit replay of prior assistant/tool/user blocks (Messages and VS Code’s normalized provider loop). An adapter must preserve both without pretending they are equivalent.
4. Subscription credentials are not portable provider credentials. ChatGPT/Codex subscription auth, Claude subscription OAuth, provider API keys, cloud-provider credentials, gateway credentials, and GitHub/Copilot billing are separate scopes. Cross-wiring subscription credentials should not be a default or inferred capability.
5. No current initial use case requires Chat Completions, and none of the inspected agent clients requires the older text Completions API. VS Code can consume Chat Completions, but Codex 0.144.1 has removed that wire API and Claude Code does not speak it. Treat Chat Completions as an optional later adapter only if a concrete endpoint requires it; do not include legacy Completions absent a separate use case.

## Surface matrix

| Client | Documented custom/provider surface | Required agent-facing wire contract | Tool and continuation shape | Credential boundary |
|---|---|---|---|---|
| Codex CLI, IDE, app-server clients | `[model_providers.<id>]` with base URL, headers, environment or command-backed auth, and provider capabilities | Current tagged source accepts `responses` only | Responses tools plus optional namespace extension; SSE; `previous_response_id` and/or explicit input history | ChatGPT sign-in or OpenAI API key for OpenAI; custom-provider credential is separate; API use is billed separately from ChatGPT |
| Claude Code | Anthropic Messages gateway, Bedrock, Vertex, Foundry, or Claude Platform on AWS | Streaming Messages or the selected cloud dialect | Flat named tools, `tool_use`/`tool_result` blocks, explicit history replay; MCP names use `mcp__server__tool` | Subscription OAuth is a Claude-specific path; gateway/API/cloud credentials take precedence and bill their owner |
| VS Code model providers | Built-ins, provider extensions, or Custom Endpoint | Per-model Chat Completions, Responses, or Messages | VS Code-normalized message/tool parts; provider adapter owns wire conversion; Custom Endpoint Responses can use `previous_response_id` | BYOK is provider-key/direct billing; Copilot-hosted models and third-party agents use GitHub/Copilot auth and policy |
| ChatGPT/Codex app, public seam | Shared Codex configuration and Codex app-server protocol are public; GUI parity for every provider setting is not promised | Upstream inference remains the selected Codex provider’s Responses contract; app-server is a separate JSON-RPC client protocol | app-server exposes thread/turn/item lifecycle, streamed notifications, resume, and fork | Public docs do not turn app login or gated GUI features into a portable provider/auth contract |

## Codex and the public ChatGPT/Codex app seam

Codex custom providers own the base URL, headers, authentication source, and wire API. Configuration supports environment-backed and command-backed credentials; `requires_openai_auth` is specifically for an OpenAI-authenticated proxy, not a generic signal that any subscription token may be reused. OpenAI documents ChatGPT sign-in as subscription access and API keys as usage-based Platform access, with API usage billed separately from ChatGPT. Enterprise access tokens are documented for trusted, non-interactive Codex local workflows; general OpenAI API calls should use Platform API keys. [Custom model providers](https://learn.chatgpt.com/docs/config-file/config-advanced#custom-model-providers) · [Codex authentication](https://learn.chatgpt.com/docs/auth)

The current tagged Codex 0.144.1 source has one accepted `WireApi` variant: `Responses`. Deserializing `chat` returns a removal error. This is stronger and more current than older prose that described Chat Completions as deprecated. [Codex `WireApi` source](https://github.com/openai/codex/blob/rust-v0.144.1/codex-rs/model-provider-info/src/lib.rs#L50-L81)

Public Responses function tools are flat objects (`type`, `name`, `description`, `parameters`, `strict`); the model emits `function_call` items and the caller returns `function_call_output`. Streaming is SSE with item and argument-delta events followed by a terminal response event. Conversation continuation can use `previous_response_id` or explicit conversation/input items. [Function calling](https://developers.openai.com/api/docs/guides/function-calling) · [Streaming Responses](https://developers.openai.com/api/docs/guides/streaming-responses) · [Conversation state](https://developers.openai.com/api/docs/guides/conversation-state)

Codex adds a provider capability for namespaced tools. Its default capability enables namespaces; the tool planner emits namespace containers with function children and only flattens/removes that representation when the provider capability says namespaces are unsupported. A provider that merely implements the public flat Responses function schema can therefore still require a Codex-specific namespace adapter. [Provider capability](https://github.com/openai/codex/blob/rust-v0.144.1/codex-rs/model-provider/src/provider.rs#L30-L47) · [Namespace capability gate](https://github.com/openai/codex/blob/rust-v0.144.1/codex-rs/core/src/tools/spec_plan.rs#L263-L270) · [Namespace tool construction](https://github.com/openai/codex/blob/rust-v0.144.1/codex-rs/core/src/tools/spec_plan.rs#L514-L555)

Codex app-server is a different boundary from model inference: it is a JSON-RPC protocol for rich clients such as the VS Code extension, exposing authentication, threads, turns, streamed items, approvals, resume, and fork. The desktop app, CLI, and IDE share Codex configuration layers, but public documentation does not establish GUI support parity for every custom-provider field or permit bypassing product login/feature gates. [Codex app-server](https://learn.chatgpt.com/docs/app-server) · [Codex configuration](https://learn.chatgpt.com/docs/config-file)

Adjacent boundary: task `019f5eb4-f85f-7d42-9b07-7da3044f41b1` owns ChatGPT login/feature gating, the disposable copied-app route, sandboxing, retained evidence, and native GUI validation. This research neither duplicates nor supersedes that work; it only records the public provider and app-server contracts it can rely on.

The current `responses-adapter` repository seam already demonstrates the narrow adaptation problem: an authenticated loopback Responses gateway with strict inbound/upstream credential separation, optional namespace flatten/reconstruction, bounded `previous_response_id` mapping, SSE behavior, and fail-closed limits. That is evidence about the existing seam, not authorization to extract, rename, or broaden it. [Repository README at the inspected checkpoint](https://github.com/nisavid/agents/blob/d7a4b62bc835012873aca95f2463b1e4e8c02c00/tooling/responses-adapter/README.md)

## Claude Code

Claude Code officially supports gateways exposing Anthropic Messages (`/v1/messages`, optionally `/v1/messages/count_tokens`), Amazon Bedrock, Google Vertex, Microsoft Foundry, or Claude Platform on AWS. `ANTHROPIC_BASE_URL` selects the Messages surface; provider-specific settings select cloud dialects. Optional model discovery is limited to Anthropic-format gateways and model IDs beginning with `claude` or `anthropic`. Claude Code does not document Responses or Chat Completions as client protocols. [Gateway overview](https://code.claude.com/docs/en/llm-gateway) · [Gateway protocol](https://code.claude.com/docs/en/llm-gateway-protocol)

Anthropic explicitly states that it does not support routing Claude Code to non-Claude models through a gateway. That is an explicit support boundary. A translating gateway may be technically imaginable, but the arrangement is outside the documented product contract and is additionally gated by dialect, model discovery, tool semantics, and authentication. [Other gateways](https://code.claude.com/docs/en/llm-gateway)

Messages tools are flat definitions with `name`, `description`, and JSON Schema `input_schema`. Claude returns `tool_use` content blocks; the caller sends matching `tool_result` blocks at the start of the next user message. Streaming is SSE with message/content-block events and partial-JSON tool-input deltas. Messages is stateless: continuation normally resends history, while a server-tool `pause_turn` response is returned as history for continuation. There is no documented `previous_response_id` equivalent. [Define tools](https://platform.claude.com/docs/en/agents-and-tools/tool-use/define-tools) · [Handle tool calls](https://platform.claude.com/docs/en/agents-and-tools/tool-use/handle-tool-calls) · [Streaming Messages](https://platform.claude.com/docs/en/build-with-claude/streaming) · [Server tools](https://platform.claude.com/docs/en/agents-and-tools/tool-use/server-tools)

Claude Code’s MCP names are model-visible as `mcp__{server_name}__{tool_name}`. Gateways must preserve Anthropic version/beta headers, and Claude Code requires inference responses to stream rather than buffer. Tool search is normally disabled for a non-first-party custom base URL because many proxies do not preserve its tool-reference blocks. These are capability-level adapter concerns, not cosmetic naming differences. [Claude Code MCP](https://code.claude.com/docs/en/mcp) · [Gateway protocol](https://code.claude.com/docs/en/llm-gateway-protocol)

Authentication distinguishes Claude subscription OAuth, Console API keys, cloud credentials, and gateway credentials. A gateway bearer or API/cloud credential takes precedence over saved subscription login and bills the credential owner. Setting only `ANTHROPIC_BASE_URL` can pass a saved Claude login through a gateway to Anthropic if required OAuth beta behavior is preserved; that does not make the OAuth token a general third-party provider credential. [Claude Code authentication](https://code.claude.com/docs/en/authentication) · [Gateway connection](https://code.claude.com/docs/en/llm-gateway-connect)

Anthropic’s current support material creates a narrow exception that must not be generalized: Agent SDK, `claude -p`, and qualifying third-party-app usage can presently draw from subscription usage through the documented Agent SDK authentication path. Separate account-auth guidance prohibits arbitrary third-party tools from routing traffic against Claude subscription limits unless Anthropic has explicitly enabled that tool. Therefore an Agent-SDK integration may be supported while a raw Codex-to-Messages gateway using a Claude subscription token remains undocumented and outside the supported credential model. [Agent SDK with a Claude plan](https://support.claude.com/en/articles/15036540-use-the-claude-agent-sdk-with-your-claude-plan) · [Claude account authentication](https://support.claude.com/en/articles/13189465-log-in-to-your-claude-account)

## VS Code and GitHub Copilot

VS Code has the broadest documented provider abstraction. BYOK can power Chat, agents, and utility tasks without GitHub sign-in or a Copilot plan. Built-in providers include Anthropic and OpenAI; extensions can contribute providers; Custom Endpoint can select Chat Completions, Responses, or Anthropic Messages per model. Agent availability requires tool-calling capability. BYOK does not replace Copilot-only inline completions, semantic search, embeddings, or related services, and organizations can disable it. [AI language models in VS Code](https://code.visualstudio.com/docs/agent-customization/language-models)

A provider extension owns model discovery, token counting, credentials, and conversion between VS Code’s normalized messages and its upstream API. It streams normalized text, tool-call, and tool-result parts. Tools have unique names, descriptions, and JSON Schema `inputSchema`; calls carry a request-local call ID. Extension tool names cannot use the reserved `copilot_` or `vscode_` prefixes, and the Custom Endpoint Responses adapter reserves `tool_search`. At the normalized layer, continuation is explicit replay of prior messages and call/result parts. Custom Endpoint may additionally use Responses `previous_response_id`; zero-data-retention mode disables that optimization and sends `store: false`. [Language Model Chat Provider API](https://code.visualstudio.com/api/extension-guides/ai/language-model-chat-provider) · [Language Model Tool API](https://code.visualstudio.com/api/extension-guides/ai/tools) · [Model configuration reference](https://code.visualstudio.com/docs/agent-customization/language-models)

VS Code also documents a distinct third-party-agent surface: Claude Agent and OpenAI Codex can run through their agent harnesses and, for eligible plans/configurations, use GitHub Copilot authentication and billing. This is a GitHub-mediated integration, not export or reuse of an Anthropic or OpenAI subscriber credential. [Third-party agents in VS Code](https://code.visualstudio.com/docs/agents/agent-types/third-party-agents)

The credential split is explicit. BYOK uses a provider API key/custom endpoint and the provider’s billing. Copilot-hosted models and Copilot-only services use GitHub/Copilot authentication, policy, and billing. GitHub’s Copilot SDK BYOK similarly bypasses Copilot authentication and talks to the chosen provider directly. No first-party VS Code/GitHub documentation exposes Claude subscription OAuth or ChatGPT/Codex subscription tokens as BYOK credential types. [GitHub Copilot SDK BYOK](https://docs.github.com/en/copilot/how-tos/copilot-sdk/auth/byok)

## Cross-wiring classification

| Desired route | Supported? | Technically gated? | Credential status |
|---|---|---|---|
| Codex-subscription GPT model in Claude Code | No: Anthropic explicitly does not support non-Claude models in Claude Code gateways | Yes: Messages/cloud dialect, discovery filter, tool/result grammar, streaming, and auth all differ | Claude Code documents no ChatGPT/Codex subscription credential input; OpenAI documents ChatGPT auth for Codex and Platform keys for general API use |
| Claude-subscription model in Codex | No documented raw Codex provider path | Yes: current Codex requires Responses; Claude exposes Messages; tools, replay/continuation, and SSE differ | Arbitrary subscription-token routing is outside Anthropic’s supported credential model and can be prohibited; the documented Agent SDK exception does not establish raw Codex gateway support |
| OpenAI API-key model behind a translator for Claude Code | Technically possible in principle, but explicitly unsupported by Claude Code for non-Claude models | Yes: a full Responses-to-Messages/tool/stream translator is required | Separately API-billed; not a Codex subscription route |
| Anthropic API-key model behind a Responses adapter for Codex | Technically possible in principle; not a native Codex-to-Anthropic configuration | Yes: a full Messages-to-Responses/tool/continuation/stream adapter is required | Separately API-billed; not a Claude subscription route |
| GPT or Claude in VS Code BYOK | Yes, with provider API keys or compatible endpoints | Provider adapter/configuration required; agent mode requires tool calling | Direct provider billing, independent of Copilot subscription |
| Claude Agent or OpenAI Codex via eligible GitHub Copilot integration | Yes, where plan/product requirements are met | Uses the documented third-party-agent harness | GitHub-mediated Copilot auth/billing; no provider subscription credential is exported |

“Prohibited” should be used narrowly. Anthropic publishes an express boundary against arbitrary third-party routing of subscription usage, with documented exceptions such as the Agent SDK path. OpenAI’s public material establishes product and credential scope but, in the sources reviewed here, does not provide an equivalent categorical terms statement about every conceivable cross-client experiment. For OpenAI, the accurate classification is unsupported/undocumented and technically gated unless a more specific policy source is identified.

## Adaptation requirements exposed by the evidence

These are requirements implied by the inspected contracts, not a settled design:

- Make wire dialect a first-class capability: Responses, Messages, and any deliberately supported cloud dialect. Do not treat “OpenAI-compatible” as sufficient.
- Model tool capabilities explicitly: flat vs namespace tools, name constraints, MCP-visible names, schema dialect, call IDs, result placement, server tools, and tool-search support.
- Model continuation explicitly: opaque response ID, explicit history replay, paused server-tool response, retention constraints, and restart/recovery behavior.
- Model streaming explicitly: event dialect, partial JSON assembly, terminal/error events, heartbeat/tolerance behavior, and whether buffering is forbidden.
- Type credentials by issuer, product, purpose, and billing scope. Never infer interchangeability from the fact that two products expose the same model family.
- Keep Chat Completions outside the initial agent-focused core unless a concrete endpoint justifies it. VS Code can adapt it at its edge; neither current Codex nor Claude Code requires it. The older text Completions API has no identified use case here.
